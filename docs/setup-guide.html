🚀 部署指南 從零到生產

fork 我的 repo，用你自己的 GitHub + Alpaca + Gmail 帳號， 一步步把整套自動再平衡系統跑起來。預計 90 分鐘完成。

① Fork ② Alpaca ③ Gmail ④ Secrets ⑤ accounts.json ⑥ Pages ⑦ Actions ✅ 驗證
🖥️ 最簡單的方式：用桌面 App（推薦） 下載 TradingAdmin 桌面程式，不用碰命令列、不用手動設 Secrets、不用手編 accounts.json。 它用 GitHub API 直接幫你管理帳戶、券商金鑰、cron 排程與日誌；你只要在 App 裡：
  1. 登入 GitHub（gh auth login，App 會引導）
  2. 在「帳戶」分頁新增帳戶 → 填券商金鑰（App 直接幫你寫進 GitHub Secrets）
  3. 在「總覽」分頁設定 email App 密碼（EMAIL_PASSWORD）
下載：github.com/itemhsu/tradingadmin/releases/latest （macOS DMG）。
下面的「手動命令列」步驟僅供進階使用者或想了解底層的人參考——用 App 的話可略過步驟 ③④⑤。

📋 系統架構

這套系統做什麼，由哪幾塊組成

每個美股交易日收盤後，GitHub Actions 自動跑：

  1. 讀 accounts.json 列出的所有帳戶
  2. 每個帳戶依其指派的 策略 JSON（如 top10 / d2p2t6 / mom_6m_t20）計算當日應有持股
  3. 對比 Alpaca 當前持倉，下「最小量換股」訂單
  4. 生成 dashboard JSON 並推到 GitHub Pages
  5. 透過 Gmail SMTP 寄每日報告到你的 email

三個 repo（公開 + 私有混合）

Repo用途可見性
tech-rebalance（主 repo） 所有 Python 程式碼 + Actions workflow + 策略 JSON + 持倉狀態 建議 Private（含交易紀錄）
tech-rebalance-dashboard GitHub Pages 入口，顯示績效、回測、log 必須 Public（免費 plan Pages 要求）
💡 為何兩個 repo？ GitHub 免費 plan 不支援 Private repo 的 Pages。我們把 Python 程式（含 API key 痕跡）放 Private，把純 HTML/JSON dashboard 放 Public，兩者透過 Actions 同步。

①Fork repo

⏱️ 5 分鐘 · 把我的程式 fork 到你帳號下

主 repo

# 1. 登入你的 GitHub
gh auth login

# 2. Fork 兩個 repo 到你的帳號
gh repo fork itemhsu/tech-rebalance --clone=true
gh repo fork itemhsu/tech-rebalance-dashboard --clone=true

# 3. 進到主 repo 目錄
cd tech-rebalance

把主 repo 設成 private（保護你的交易紀錄）

gh repo edit YOUR_USERNAME/tech-rebalance --visibility=private
💡 fork 後的 commits 仍會推到我的 upstream 嗎？ 不會。Fork 是獨立的副本。除非你自己手動 PR 否則 commits 只在你 repo。

②申請 Alpaca 帳號

⏱️ 20 分鐘（含身分驗證）· 取得 API Key 與 Secret
  1. 到 https://alpaca.markets → Sign Up
  2. 填基本資料 → 上傳身分證件（線上拍照）→ 等審核（通常 1-2 小時）
  3. 登入 Dashboard → 左側選單 Paper Trading（不是 Live）
  4. 右上角 View API Keys → Generate New Key
  5. 記下 API Key ID 與 Secret Key（Secret 只顯示一次！）

每個策略要不同 Alpaca 帳戶嗎？

Paper Trading 一個帳戶就夠（虛擬資金 $100K）。實單要分開：每個策略獨立帳戶才能追蹤各自績效。Alpaca 一個 email 可以開多個 sub-account。

✅ 建議初次設定 先用 Paper Trading 1 個帳戶，跑通流程 + 看 1-2 個月績效，再考慮開實單帳戶。
⚠️ Secret Key 安全 永遠不要把 Secret 貼到 chat / Slack / 程式碼裡。只能放 GitHub Secrets。

③設定 Gmail 寄信

⏱️ 5 分鐘 · 每日報告寄到你的 email

系統用 Gmail SMTP 寄信，只需要一組 Gmail App Password（應用程式密碼），不需要第三方服務、不需信用卡、零成本。

  1. 先確認該 Gmail 帳號已開啟 兩步驟驗證（App Password 的前提）：
    • 到 Google 帳戶 → 安全性
    • 找「兩步驟驗證」→ 若未開啟先開啟
  2. 產生 App Password：
    • 到 myaccount.google.com/apppasswords
    • App 名稱填任意（例如 tech-rebalance）→ 建立
    • 記下顯示的 16 碼密碼（格式 abcd efgh ijkl mnop）
  3. 把 16 碼去掉空格後存成 GitHub Secret EMAIL_PASSWORD（下一步詳述）。
🔑 App Password ≠ 你的 Gmail 登入密碼 一定要用上面產生的 16 碼專用密碼。直接用一般登入密碼會被 Google 擋下 （535 BadCredentials）。
📧 寄件人與收件人 寄件人 = App Password 所屬的 Gmail（EMAIL_SENDER）。 收件人由 accounts.json 每個帳戶的 email_recipients 決定， 可一個帳戶寄給多人。沒設 EMAIL_SENDER 時自動用收件人當寄件人。

④設定 GitHub Secrets

⏱️ 10 分鐘 · 把 API keys 安全地放進 GitHub

到你的 repo 設定頁：github.com/YOUR_USERNAME/tech-rebalance/settings/secrets/actions

必填 Secrets

Secret 名稱值來自說明
ACC1_ALPACA_KEY 步驟 ② Alpaca 第一個帳戶的 API Key ID
ACC1_ALPACA_SECRET 步驟 ② Alpaca 第一個帳戶的 Secret Key
ACC1_BASE_URL 固定值 https://paper-api.alpaca.markets（Paper）
實單改 https://api.alpaca.markets
EMAIL_SENDER 步驟 ③ 的 Gmail（App Password 所屬帳號） e.g. you@gmail.com，同時是 SMTP 登入帳號
EMAIL_PASSWORD 步驟 ③ 產生的 16 碼 App Password 去掉空格，e.g. abcdefghijklmnop
💡 用桌面 App 的話，這一步全自動 在 TradingAdmin App 的「帳戶」分頁新增帳戶時直接填券商金鑰，App 會幫你寫進 ACC{N}_ALPACA_KEY / ACC{N}_ALPACA_SECRET； email 密碼在「總覽」分頁設定 EMAIL_PASSWORD。你不需要記任何 Secret 名稱。
📮 收件人在哪設？ 收件人不是 Secret，而是寫在 accounts.json 每個帳戶的 email_recipients 陣列（步驟 ⑤）。這樣不同帳戶可寄給不同人。

多帳戶?再加 ACC2 / ACC3 ...

每多一個帳戶就加三個 Secret：ACC2_ALPACA_KEY、ACC2_ALPACA_SECRET、ACC2_BASE_URL。

用 gh CLI 一次設好（手動方式） 
gh secret set ACC1_ALPACA_KEY      --body "你的 Alpaca key"
gh secret set ACC1_ALPACA_SECRET   --body "你的 Alpaca secret"
gh secret set ACC1_BASE_URL        --body "https://paper-api.alpaca.markets"
gh secret set EMAIL_SENDER         --body "you@gmail.com"
gh secret set EMAIL_PASSWORD       --body "abcdefghijklmnop"
📊 Dashboard 自動更新的金鑰由維護者設定（你不用碰） 讓主 repo 把 dashboard 推到公開 Pages repo 的金鑰是 PAGES_TOKEN， 屬一次性基礎建設、由系統維護者設定一次即可，一般使用者完全不需要處理。

⑤編輯 accounts.json

⏱️ 10 分鐘 · 告訴系統有哪些帳戶、各跑什麼策略

編輯 accounts.json（位於 repo 根目錄）。最簡單的單帳戶配置：

{
  "accounts": [
    {
      "id": "1",
      "strategy": "top10",
      "label": "帳戶 #1 (TOP10 月度)",
      "enabled": true,
      "alpaca_secret_prefix": "ACC1",
      "data_dir": "data/1"
    }
  ]
}

多帳戶配置範例

{
  "accounts": [
    {
      "id": "1", "strategy": "top10",
      "label": "TOP10 月度",
      "enabled": true, "alpaca_secret_prefix": "ACC1", "data_dir": "data/1"
    },
    {
      "id": "2", "strategy": "d2p2t6",
      "label": "D2P2T6 多元",
      "enabled": true, "alpaca_secret_prefix": "ACC2", "data_dir": "d2p2t6/data/1"
    },
    {
      "id": "3", "strategy": "mom_6m_t20",
      "label": "MOM-6m 動能",
      "enabled": true, "alpaca_secret_prefix": "ACC3", "data_dir": "data/3"
    }
  ]
}

欄位說明

id帳戶識別字（用於檔案路徑、log）
strategy對應 strategies/{strategy}.json 檔名
label顯示在 email、dashboard 的人類可讀名稱
enabledfalse 時暫停執行（保留設定）
alpaca_secret_prefix對應 GitHub Secrets {prefix}_ALPACA_KEY
data_dir持倉狀態 + log 寫入路徑

✅ 驗證：先跑 DRY-RUN 確保流程通

⏱️ 10 分鐘 · 不下單，只模擬整個流程

手動觸發 Daily All Accounts（dry-run 模式）

  1. 到主 repo → Actions → Daily All Accounts
  2. 點 Run workflow 按鈕
  3. dry_run 輸入 true → Run workflow
  4. 等 2-3 分鐘執行

驗證 checklist

  • workflow 變綠（所有 step 成功）
  • log 中看到 [INFO] trader: 無再平衡訂單需要執行（dry-run + 空帳戶）
  • Alpaca dashboard 沒有任何訂單（確認 dry_run 生效）
  • email 收到一封「[DRY RUN]」開頭的測試報告（需設好 EMAIL_SENDER + EMAIL_PASSWORD 並在 accounts.json 填 email_recipients）

用 gh CLI 監看

gh run watch                              # 跟著最新 run 的 log
gh run view --log-failed                  # 失敗 step 的 log
gh run list --workflow=daily_all_accounts.yml --limit 3
✅ dry-run 通了 = 環境設好了 可以放心讓排程自動跑，明天早上就會收到第一封實際報告（dry_run=false）。

🔧 新增 / 修改策略

⏱️ 5-10 分鐘 · 純 JSON 設定，零 Python 修改

系統用 v3 Schema 描述策略，加新策略只要寫一個 JSON 檔。

範例：新建「科技 50 + 9m 動能」策略

# 1. 複製模板
cp strategies/mom_6m_t20.json strategies/mom_9m_t50.json

# 2. 編輯關鍵欄位：
{
  "id": "mom_9m_t50",
  "meta": {"name": "MOM-9m 科技 50 動能"},
  "selection": {
    "method": "weighted_percentile",
    "n": 10, "watchlist_top_n": 50, "watchlist_metric": "market_cap"
  },
  "ranking": {
    "factors": [
      { "field": "price_momentum_9m", "weight": 1.0, "direction": "desc" }
    ]
  },
  "backtest": {
    "color": "#a855f7", "group": "momentum", "order": 110
  }
}

# 3. 在 accounts.json 加帳戶或修改既有帳戶
{"id": "3", "strategy": "mom_9m_t50", ...}

# 4. push
git add strategies/mom_9m_t50.json accounts.json
git commit -m "feat: add mom_9m_t50 strategy"
git push

排程下次跑時自動採用新策略。回測也會自動納入 momentum dashboard。

可用 selection methods

top_n_by_metric依單一指標排序取前 N（如純市值）
top_n_per_group分組各取 N 檔（如 D2P2T6）
factor_score多因子加權打分
weighted_percentilewatchlist 預篩 + 單因子（典型動能）
weighted_percentile_per_group分組 watchlist + 因子排序
buy_and_hold買入持有不調整（基準用）

👀 監控與除錯

怎麼確認系統健康、出錯怎麼查

每日例行檢查

  • 早上 8 點檢查 inbox：應有「每日報告」email
  • 沒收到 → 到 GitHub Actions 看排程 run 是否成功
  • NAV 變動異常 → 看 email 內的「今日交易」段
  • 有交易 → log 頁面看詳細事件（trigger / reason / 守則）

常用除錯指令

# 看最近 5 次 daily 執行狀態
gh run list --workflow=daily_all_accounts.yml --limit 5

# 看失敗 run 的 log（找特定 step）
gh run view RUN_ID --log-failed

# 手動觸發補跑（dry-run 安全測試）
gh workflow run daily_all_accounts.yml -f dry_run=true

# 看本機 trade_events（jsonl 格式可 jq 查詢）
jq 'select(.type=="ORDER_PLANNED")' data/1/trade_events.jsonl

常見錯誤

症狀原因修法
workflow 0 秒失敗 YAML 語法錯（多半是 PR merge 衝突） 用 python3 -c "import yaml; yaml.safe_load(open('.github/workflows/foo.yml'))" 驗
Alpaca 金鑰未設定 Secret 名稱與 accounts.json 的 prefix 對不上 確認 ACC{N}_ALPACA_KEY 對應 "alpaca_secret_prefix": "ACC{N}"
郵件沒寄出 535 BadCredentials EMAIL_PASSWORD 用了一般登入密碼，或沒去掉空格 用步驟 ③ 的 16 碼 App Password，存 Secret 時不含空格
策略 schema validation 失敗 JSON 不符 v3 schema python3 -c "import json,jsonschema; ..." 在本機驗
Dashboard 顯示 404 Pages 未啟用 / PAGES_TOKEN 失效 步驟 ⑥ 啟用 Pages；確認維護者已設好 PAGES_TOKEN
