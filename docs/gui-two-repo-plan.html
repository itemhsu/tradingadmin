1. 背景

GUI（tradingadmin）目前綁舊 fork 模型：精靈幫你 Fork tech-rebalance（整包引擎碼複製進你的 repo）。 但我們已驗證兩 repo 模型：使用者只有一個薄殼 Repo B（accounts.json + data/ + 薄 workflow + vendored 引擎 wheel），引擎按版本安裝。 GUI 要跟上，否則使用者得手動在 GitHub 點來點去（建 repo、放檔、設金鑰、跑）。

2. 關鍵衝擊：Repo B 不含 strategies/schemas/brokers ★

兩 repo 模型下，策略 / schema / 券商定義都在引擎 wheel 裡，不在使用者 repo。 GUI 現在有幾處直接讀使用者 repo 的這些目錄，在 Repo B 會讀到空 → 必須改資料來源。
GUI 功能現在讀哪Repo B 會怎樣改成
策略下拉選單（catalog.list_strategies）repo 的 strategies/空（在 wheel 裡）讀引擎 manifest（§6）
券商下拉（catalog.list_brokers）repo 的 brokers/空讀引擎 manifest
必填 Secret 推導（required_secrets）brokers/{id}.json讀不到manifest 內附 broker auth

4. 核心改動

4.1 精靈：「② 建立交易系統（Repo B）」取代 Fork

把我手動用 gh 做的流程做成一鍵（純 API）：

  • 建 private repo（gh repo create {user}/tech-rebalance-data --private）。
  • 放範本檔：accounts.json、data/.gitkeep、薄 daily.yml、README。
  • 下載引擎最新 wheel 放進 vendor/（從引擎 Release）。
  • workflow 初始設「手動觸發、dry_run 預設 true」（防呆）。
已具備的素材：gh_client（建 repo/設 secret）、repo_store.GhContentsStore（推檔，含巢狀路徑）。新增一支 repo_b_provisioner 服務串起來。

4.2 引擎版本管理（取代「同步上游」）

  • 顯示目前釘的版本（讀 vendor/*.whl 檔名或 daily.yml 的 @vX）。
  • 「更新引擎」按鈕：列出引擎 Release 版本 → 選一個 → 下載 wheel 換掉 vendor/ + 改 daily.yml 檔名 + commit。
  • 新增服務 engine_release：gh release list/download --repo itemhsu/tech-rebalance。

4.3 執行按鈕（新）

Overview 或新「執行」分頁加按鈕，觸發 Repo B 的 daily.yml（gh workflow run … -f dry_run= -f force=）：

  • 測試執行（dry-run）：不下單，看選股/計劃。
  • 強制再平衡：force=true（手動換股；live 時跳確認）。
  • 看執行紀錄：列 workflow runs + 點開 log（gh run view）。

4.4 排程開關

schedule_view 目前硬寫 daily_all_accounts.yml；改成讀 Repo B 的 daily.yml，提供「啟用/停用每日排程」開關（取消/加回 cron 註解）。

4.5 paper / live 切換 + 防呆

  • 帳戶頁加 environment 切換（paper / live）。
  • 切 live 或在 live 下觸發交易 → 強制確認對話框（紅字、需打字確認），呼應「真錢由你親手」。
  • 可管理「兩個 Repo B」：paper repo 與 live repo（config 存兩個 slug）。

4.6 drift 橫幅改資料來源

現在的 compat.schema_drift_warning 讀 repo 的 schemas/；Repo B 沒有。改成讀 vendor/ wheel 的版本，與 App 支援的引擎版本範圍比對（App 內建 ENGINE_SUPPORTED 集合）。

6. 引擎端配合：發佈 manifest.json

GUI 需要「這版引擎有哪些策略/券商」但 Repo B 沒有這些檔。引擎 Release 多附一個 manifest.json：

{
  "engine_version": "1.0.3",
  "data_schema": "1.0",
  "strategies": ["top10", "d2p2t6", "mom_6m_t20", ...],
  "brokers": {
    "alpaca":  { "required_env": ["{PREFIX}_ALPACA_KEY","{PREFIX}_ALPACA_SECRET"], "environments": ["paper","live"] },
    "tradier": { "required_env": ["{PREFIX}_API_KEY","{PREFIX}_ACCOUNT_ID"] }
  }
}
  • 由引擎 release.yml 自動產生（掃 strategies/ + brokers/）並附到 Release。
  • GUI 抓 manifest（gh release download manifest.json）→ 餵下拉選單與必填 secret 推導。
  • 順帶讓「策略清單」永遠跟引擎版本一致，不會有 GUI 顯示了引擎沒有的策略。

8. 分階段 + 向後相容 + 發版

向後相容（重要）

你現有的舊系統（itemhsu/tech-rebalance fork 模式、3 個 live 帳戶）還在運作。GUI 不能只支援新模式。 → GUI 啟動時偵測模式：repo 有引擎碼（runner.py）= fork 模式（沿用現有 UI）；有 vendor/ + 薄 daily.yml = repo_b 模式（新 UI）。兩種並存，逐步遷移。

一句話

GUI 改造的核心是把資料來源從「使用者 repo 的目錄」換成「引擎 manifest + vendored wheel」，並把建 Repo B / 更新引擎 / 跑 / 切 live 做成按鈕。 最大技術點是 §2（catalog/compat 來源）與 §6（引擎發 manifest）；其餘是把我手動用 gh 做的流程包成服務。向後相容靠 mode 偵測，舊 live 系統不受影響。
